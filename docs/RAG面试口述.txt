————————————————————
一、先用一句话说清楚
————————————————————

这是一个本地文档问答系统。前端上传 PDF 或 TXT；Node 网关暂存文件并管会话；Python 负责解析、切块、向量检索和调本地大模型。问答时先检索当前文档里最相关的几段，再流式生成答案。检索不是纯向量，还有词面加分和交叉编码器重排。

————————————————————
二、整体架构和两条链路
————————————————————

三层：前端（Umi）→ Node（Express，大约三千端口）→ Python（FastAPI，大约八千端口）。

本地依赖还有：
- 向量库 Chroma，持久化切块和向量
- LM Studio，提供 OpenAI 兼容接口，大约一万一千四百三十五端口
- Node 侧 SQLite，存会话、消息、文档摘要、对话记忆

离线建库：上传 → 解析成按页文本 → 切块 → BGE 向量写入 Chroma。
在线问答：对话记忆处理 → 改写检索问句 → 召回 → 打分过滤去重 → 重排 → 拼提示词 → LM Studio 流式生成。

原文件不长期保存：Node 用 multer 先落到 uploads 临时目录，处理完就删；内容靠向量库里的切块，业务信息靠 SQLite。

三、不同类型文档怎么解析（要讲清用什么抽）
————————————————————

目前只支持 PDF 和 TXT。

【TXT 怎么读】
用文件读取，按常见编码把全文读进来。整份文件当成一页文本，页码记成零，表示没有真实页概念。不做 OCR，不做版面分析。后面一律走递归字符切块。

【PDF 总流程】
按页处理，分两大阶段：先抽数字层，再按页选策略（纯数字层 / 图文混排 / 整页 OCR）。页数默认最多大约一百二十页。

————————
阶段 A：每一页「数字层文字」到底怎么抽
————————

「数字层」指 PDF 里本身可选中、可复制的那层文字，不是对着图片看出来的字。

抽取工具优先级：

1. 优先用 pdfplumber
   - 正文：对每一页调用抽取文本接口，拿到该页可复制文字
   - 表格：同一页再用 pdfplumber 抽表，把二维表转成 Markdown 表格文本，并加上「表格」标题，方便后面检索和结构切块
2. 如果环境没有 pdfplumber，或者抽失败，回退用 PyMuPDF（fitz）
   - 对每一页 get_text，抽纯文本数字层
   - 表格能力弱一些，主要保正文

同时用 PyMuPDF 估「这一页图像占页面面积多大」，作为后面是否 OCR 的依据。版面分析也主要靠 PyMuPDF：读页面文字块、图片块的几何框，不用额外深度学习版面模型。

这一阶段每页产出三样东西：数字层正文、表格 Markdown（有的话）、图像覆盖率。

————————
阶段 B：按页选三种策略，OCR 用什么
————————

OCR 引擎：RapidOCR（优先新包 rapidocr，不行再试旧的 onnxruntime 包）。  
表格增强：RapidTable，从页面图里再抽表结构，转成 Markdown；并和 OCR 行去重，减少同一内容进库两次。

判定时看两个量：数字层字数、图像覆盖率；再结合版面里有没有值得单独抠的插图。

经验阈值（口述可带上）：
- 数字层大约八十个字以上，算「字够丰富」
- 少于大约四十字，算「字很稀疏」
- 图像覆盖大约百分之三十五以上，像扫描页
- 几乎没字但图像覆盖大约百分之十五以上，也倾向整页 OCR
- 单张图占页面积大约百分之八十五以上，当近整页大图
- 很小的图标（面积比大约百分之四以下）忽略，不单独 OCR

三种策略：

第一种，digital（纯数字层）
条件：数字层够用，没有必要做插图 OCR。
做法：直接用阶段 A 抽出的文字加表格 Markdown，不跑 RapidOCR。最快，原生 PDF 质量最好。

第二种，full_ocr（整页 OCR）
条件：几乎没字的扫描页、稀疏文字加高图像占比、或近整页大图。
做法：用 PyMuPDF 把整页渲染成图片（大约两倍分辨率），交给 RapidOCR 识别全文；同时可走 RapidTable 抽表。合并时以 OCR 结果为主，再拼表格，并去掉已在表格里出现的 OCR 行。

第三种，hybrid（图文混排）
条件：正文数字层还行，但页里还有中等大小的插图，图里可能有字。
做法：
- 正文优先用 PyMuPDF 版面里的文字块；没有文字块就用整页数字层
- 对每个非整页插图区域，用 PyMuPDF 按框裁剪渲染，再 RapidOCR
- 图内结果写成「图内文字」加编号，方便后面结构切块识别
- 所有块按阅读顺序（先上后下、先左后右）拼回一页，并做行级去重，减少版面块和 OCR 重复

面试收束一句：PDF 不是整本一律 OCR。能 pdfplumber 或 PyMuPDF 抽数字层就抽；混排只对插图区域 RapidOCR；扫描页才整页渲染加 RapidOCR。

四、怎么切块（算法和参数要说清）
————————————————————

入口在切块模块，默认参数：
- 块大小大约三百个字符
- 块间重叠大约五十个字符
- 过短碎块阈值大约块大小的百分之三十，短于这个就并进相邻块
- 策略默认 auto，也可强制指定

硬原则：按页切，不跨页合并。PDF 一页是天然边界。

【策略怎么选】
- TXT：固定走递归字符切分
- PDF 某一页：如果页里出现「图内文字」「表格」标记，或 Markdown 表格行，走结构感知切分；否则走递归字符切分
所以 hybrid 或 OCR 产出的图内标记、表格 Markdown，会自然把该页带进结构切分。

【递归字符切分（recursive）】
思路对齐常见的 Recursive Character Text Splitter：
分隔符从粗到细：空段、换行、中文句号问号等、英文标点、逗号顿号、空格，最后才按长度硬切。
先尽量在语义边界切开并合并到不超过三百字；递归过程中不叠加重叠，最后再统一给下一块带上上一块大约五十字的尾巴，且最终长度不超过三百。过短碎块再合并。

【结构感知切分（structure）】
先按行扫成三类单元：普通正文、Markdown 表格、图内文字或表格标题块。
- 表格：尽量整表保留，太长按行切，避免从单元格中间切断
- 图内文字：单独成块
- 正文：按句子装箱到大约三百字，块间可带重叠
同类型过短单元可合并，但正文、表格、图注不混并。

每个块会带上：正文、页码（TXT 为０）、类型（正文或表格或图内）、本页用的切块策略。回答时不要求模型念页码。

五、向量化、写入哪里、用什么模型
————————————————————

向量模型：BAAI 的 bge-small-zh-v1.5（SentenceTransformer）。
- 文档块入库：原文直接编码，不加指令前缀
- 查询检索：先加中文检索指令「为这个句子生成表示以用于检索相关文章：」再编码
- 向量做 L2 归一化，方便相似度比较

向量库：Chroma 持久化目录，集合名 documents。按文档编号过滤，同文档重复上传先删旧向量再写。大约二十条一批 embed 再写入。中途取消会尽量回滚半成品。

业务库 SQLite：会话、消息、文档摘要、滚动记忆摘要等。原文件不归档。

文档摘要：入库后另调 LM Studio 做 map-reduce 式摘要，给界面展示；失败不影响问答。

六、用户提问之后逐步发生什么
————————————————————

【1. 对话记忆】
默认近窗大约五轮原文（配置夹在四到六轮）。更早对话用大模型压成滚动摘要，上限大约一千二百字；用「已覆盖进度」做增量压缩，不是每轮从头重压。
压缩也是调 LM Studio 同一个对话模型，提示它合并旧摘要和新挤出窗口的对话，保留事实和未决问题，去掉寒暄。

【2. 检索问句改写】
还是同一个对话模型，结合摘要和近窗，把「这个呢」改成独立可检索问句。失败回退原问题。
改写只用于检索；最终回答仍用用户原话。

【3. 向量初召回】
用 BGE 把改写问句编码，在 Chroma 里只搜当前文档。召回条数自适应：小文档几乎全召回；大文档大约按块数百分之十二、并结合最终条数的倍数估算，上限大约五十。

【4. 向量分加词面分】
把距离转成越大越好的向量分，再加上词面加分：整句命中、去标点命中、四字窗口命中率、编辑相似度等，命中高可加大约零点三到零点五五。短问打长块时特别有用。

【5. 阈值和兜底】
默认相似度阈值大约零点一五；很小的文档更松，大约零点零五；开着重排时预过滤也会再放宽一点。
滤光了仍尽量保留靠前候选；向量查询异常空了，还可退回全量加词面打分。原则是库里有内容尽量别问空。

【6. 去重】
重叠切块很常见，文本相似度大约零点七二以上会去重，保留更长更完整的那段，减轻复读。

【7. 重排】
模型：BAAI 的 bge-reranker-base，交叉编码器。对「问题加候选段落」成对打分，经 sigmoid 后排序，截断到最终条数。默认只排序不硬滤空。未加载好或失败时退回向量顺序。

【8. 最终给模型几段】
大约两到五段，按整篇文档总块数自适应：很小大约两段，中等三四段，很大最多五段。不是召回多少就塞多少。

【9. 拼提示词并生成】
系统要求依据文档片段回答、要点不重复、不标页码。再放摘要、近窗历史、编号后的片段、用户原问题。
生成模型：LM Studio 上的本地量化对话模型（代码里名称是 123@q4_k_m）。温度大约零点三，最长大约八百 token，带抑制重复的惩罚，真流式输出。

————————————————————
七、怎么处理幻觉
————————————————————

从检索和生成两边压。

检索侧：只搜当前文档；词面加分加重排提高相关性；最终只留两到五段干净上下文；重叠去重。
提示词侧：明确根据文档片段综合回答，不要逐段复述、不要复读扩写；回答用原问题。
生成侧：温度偏低、长度上限、抑制重复。

没入库直接提示上传。库里有但问偏时，当前更偏向给近邻片段保可用性，而不是一律拒答——这是取舍。若业务更怕幻觉，可再加「相关度不够就明确说未提及」的硬门槛。

复读和事实幻觉要分开说：复读用去重、提示词、生成惩罚一起压。

————————————————————
八、怎么处理上下文
————————————————————

三块分开管：

文档上下文：三百字切块、召回大池、精排后只留两到五段；提示词里片段和问题分区，提醒合并重叠、勿逐段抄。
对话上下文：近窗原文加滚动摘要（大约一千二百字上限），增量更新。
检索上下文：只用改写问句去搜；生成仍看原问题。

本地小模型窗口有限，token 预算优先留给文档片段，聊天史不能无限膨胀。

九、系统一共用到哪些模型
————————————————————

1. 对话大模型：LM Studio 本地模型 123@q4_k_m  
   用途：最终回答、对话摘要压缩、文档摘要、检索问句改写

2. 向量模型：bge-small-zh-v1.5  
   用途：切块入库、问题编码检索

3. 重排模型：bge-reranker-base  
   用途：召回后精排

4. OCR：RapidOCR  
   用途：扫描页整页识别、插图区域识别

5. 表格结构（可选）：RapidTable  
   用途：从页面图抽表，配合 OCR

解析抽字工具不算「大模型」，但面试要说清：数字层靠 pdfplumber 优先、PyMuPDF 兜底和版面几何。

————————————————————
十、还可以补充的工程点
————————————————————

文档隔离：按文档编号检索，不串库。
失败降级：摘要失败仍可问；重排未就绪退回向量序；改写失败用原问。
取消：上传解析和问答支持断开即停；半写入向量尽量回滚。
预热：启动后后台预热向量模型和重排模型，避免首问卡加载。
解析质量也是 RAG 质量：很多「胡答」其实是字没抽对、表被切断、图内字丢了。
评估：可用固定问句集看召回和精排是否命中金标内容。

十一、两分半串讲稿
————————————————————

我们做本地 RAG。上传后 Node 暂存 uploads，Python 解析：TXT 整篇读取；PDF 先用 pdfplumber 抽数字层和表，不行就 PyMuPDF，再按页看字数和图像占比，决定纯数字层、插图区域 RapidOCR，还是整页渲染 RapidOCR。

切块按页，大约三百字五十重叠。纯文递归切；有表和图内标记的页结构切。BGE-small 非对称编码进 Chroma。

提问时近五轮加滚动摘要管对话；改写问句只用于检索。Chroma 召回后加词面分、去重，bge-reranker 精排到两到五段，交给 LM Studio 流式回答。防幻觉靠检索准、上下文干净、提示词 grounded、生成参数压着。

十二、追问答法（细化）
————————————————————

问：数字层文字具体用什么抽？
答：优先 pdfplumber 抽正文和表；没有或不行就 PyMuPDF get_text。图像占比和版面框也主要靠 PyMuPDF。OCR 才上 RapidOCR。

问：为什么不是纯向量？
答：短问对长块向量分不稳，所以加词面加分；再用交叉编码器精排。

问：召回很多为什么只给模型几段？
答：召回服务重排候选池；进模型要控长度和噪声，按文档规模大约两到五段。

问：记忆怎么压？
答：也是 LM Studio 同一个对话模型做滚动摘要，上限大约一千二百字，近窗大约五轮原文，增量合并。

问：一共几个模型？
答：对话一个、向量一个、重排一个、OCR 一个，表格结构可选；外加 pdfplumber 和 PyMuPDF 做数字层抽取。

问：原文件存哪？
答：Node 的 uploads 临时目录，处理完删除；长期靠 Chroma 切块和 SQLite 会话摘要。

问：怎么减少幻觉？
答：文档隔离、重排、少而准上下文、去重、提示词要求依据片段、低温限长；并说明弱相关兜底是可用性取舍。
